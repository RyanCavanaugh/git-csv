Responding to some of the earlier concerns:

Evidence of lots of existing usage

This approach probably isn't being used right now because it doesn't work. But pretty much any existing codebase that defines an interface and uses interface Child extends Parent would arguably be evidence for this use case. It's just that there aren't as many JSDoc users yet; but the use case is the same. With some of Rich Harris's philosophies on JSDocs getting more popularized, we may see this more "explicitly" in libraries in the future.

Why Not .d.ts Files?

This approach isn't as convenient when it comes to library development. In normal TS land, it makes sense to export related types and functions from the same .ts file. So naturally, it would make sense to do the same thing for a .js file. Unfortunately, we cannot; and there are caveats that come from using a separate .d.ts file.

For end users, it's inconvenient to write

import { myFunction } from "my-library";
import type { myTypeRelatedToFunction } from "my-library/some-types-file";

for every JS file that has complex types. (Barrel files are not a silver bullet here because not all consumers have access to tree shaking.)

At the same time, for library maintainers, it's inconvenient to write

/library
  my-js-file.js
  my-js-file-types.d.ts

every time complex types come into play. But beyond the minor file-structure inconvenience, it makes it makes the build process more difficult. Where do the types go? If some types are in the .d.ts file while the actual implementations (functions, classes, etc.) are in the js file, then there's a split in where the related types exist. Do we create a new .d.ts file that exports types from both files? How would maintainers automate that process and make it reliable? (Does that dilemma make sense?) Developers using .ts files never have to answer these questions (if they don't want to).

⊚ #51620 Assignability of covariant types with bivariant methods now fails in certain intersection types

@RyanCavanaugh assigned on 12/1/2022
@ahejlsberg 3 days ago

This one is interesting. The immediate cause of the change in behavior is #51140. However, that PR simply improved our checking enough to reveal a deeper issue that has existed ever since #15104 was introduced in TS 2.4. In that PR we started checking callback parameters covariantly instead of bivariantly:

type Covar<T> = { setCallback(cb: (value: T) => void): void }

declare let cu: Covar<unknown>;
declare let cs: Covar<string>;
cu = cs;  // Ok
cs = cu;  // Error, but wasn't before #15104

The specific change in #15104 was to detect parameters of single-signature function types and exclude them from the bidirectional checking we otherwise do for parameters of methods. But there's a subtle inconsistency that can arise from this:

type Bivar<T> = { set(value: T): void };

declare let bu: Bivar<unknown>;
declare let bs: Bivar<string>;
bu = bs;  // Ok
bs = bu;  // Ok

declare let bfu: Bivar<(x: unknown) => void>;
declare let bfs: Bivar<(x: string) => void>;
bfu = bfs;  // Ok
bfs = bfu;  // Ok

Above, Bivar<T> is bivariant in T and we measure it's variance accordingly. That means that two instantiations of Bivar<T> are related as long as the type arguments for T are related in one direction or the other. But, in the second example above, we also have a rule that says methods with parameters of single-signature function types should relate covariantly, and the set method qualifies as such in an instantiation of Bivar<T> where T is a single-signature function type. Above, bivariant checking wins because we trust the measured variance for T, but when relating structurally we get a different outcome:

type Bivar1<T> = { set(value: T): void };
type Bivar2<T> = { set(value: T): void };

declare let b1fu: Bivar1<(x: unknown) => void>;
declare let b2fs: Bivar2<(x: string) => void>;
b1fu = b2fs;  // Ok
b2fs = b1fu;  // Error

That's a problem. And the root cause of this issue. Here's the effect with a Set-like type:

type SetLike<T> = { set(value: T): void, get(): T }

declare let sx: SetLike1<(x: unknown) => void>;
declare let sy: SetLike1<(x: string) => void>;
sx = sy;  // Error
sy = sx;  // Ok

type SetLike1<T> = { set(value: T): void, get(): T }
type SetLike2<T> = { set(value: T): void, get(): T }

declare let s1: SetLike1<(x: unknown) => void>;
declare let s2: SetLike2<(x: string) => void>;
s1 = s2;  // Error
s2 = s1;  // Error, but shouldn't be

To fix this we need to slightly amend the callback checking rules such that we only special case parameters of single-signature function types when they have single-signature function types for all possible instantiations of the containing signature. In other words, no special callback checking when the pattern arises due to instantiation of a parameter with a naked generic type.

⊚ #31267 Add support for @inheritdoc

👍10
@hahanein on 6/27/2023

It is common in React to group logical components as a single object.
A straight-forward way to accomplish this using TypeScript is the following:

/** Groups {@link Item} components. */
const Group: FC<PropsWithChildren> = props => <div>{children}</div>;
/** An item of the {@link Group} component. */
const Item: FC<PropsWithChildren> = props => <div>{children}</div>;

export default Object.assign(Group, {Item});

And then you use it like so:

<Group>
  <Group.Item>A</Group.Item>
  <Group.Item>B</Group.Item>
</Group>

However, this pattern swallows the documentation. With @inheritDoc, as described in TSDoc, you could do the following:

/** Groups {@link Item} components. */
const Group: FC<PropsWithChildren> = props => <div>{children}</div>;
/** An item of the {@link Group} component. */
const Item: FC<PropsWithChildren> = props => <div>{children}</div>;

/** {@inheritDoc Goup} */
export default Object.assign(Group, {
  /** {@inheritDoc Item} */
  Item
});

Without it, there is no work-around except for copy-pasting the documentation.

The computed signature of the first callback here is:

(project: number) => Promise<{ requiredProperty: number; optionalProperty2: number; }> | { requiredProperty: number; optionalProperty1: number; }

and that is being used as an inference source for this signature:

(value: number): TResult1 | PromiseLike<TResult1>

On top of that... non-naked targets are preferred and that's why the inference gets fixed with { requiredProperty: number; optionalProperty2: number; }.

⊚ #44913 Promise.catch `null` vs `null as null`

@RyanCavanaugh milestoned on 2/1/2023
@rbuckton 5 days ago

I think this happens because in the absence of a type annotation, null gets inferred as any, even under strictNullChecks.

let x = null;  // type: any
let y = null as null;  // type: null

That's not quite true. When outside of --strictNullChecks there are two null types: A non-widening null type and a widening null type. The widening null type is returned when null is used as an expression, while the non-widening null type is returned when null is used as a type. At any point where we widen (such as when we widen "foo" to string), the widening null is either removed from any union it is contained in, or is converted to any, while the non-widening null is only removed from unions if there are any other non-widening types. When inside of --strictNullChecks, there is only ever one null type, which is the non-widening null type. The same is also true for undefined.

The example you show here isn't quite correct, as the type you see when you hover over x isn't the inferred type, it is the type resulting from control flow analysis:

// @strictNullChecks: true
let x = null;
//  ^?  let x: any

   x;
// ^?   let x: null

Playground

In the example from the OP, you can see that both number and null are preserved in --strictNullChecks:

// @strictNullChecks: true
export function f1()
{
	return Promise.resolve(1).catch(e => null)
}

export function f2()
{
	return Promise.resolve(1).catch(e => null as null)
}

// @ts-expect-error
f1().then(v => v.split('')) // <====== should is `number` but `any`
//        ^? (parameter) v: number | null

// @ts-expect-error
f2().then(v => v.split(''))
//        ^? (parameter) v: number | null

Playground

The reason why you see any is due to null being the only possible return value of .catch. As a result, it widens to any as the type escapes the body. You can see that here, though I had to replace the arrow functions with function expressions so I could have something to point the ^? comment at:

// @strictNullChecks: false
export function f1()
{
	return Promise.resolve(1).catch(function(e) { return null; })
        //                              ^? (local function) (e: any): any
}

export function f2()
{
	return Promise.resolve(1).catch(function (e) { return null as null; })
        //                              ^? (local function) (e: any): null
}

// @ts-expect-error
f1().then(v => v.split('')) // <====== should is `number` but `any`
//        ^? (parameter) v: any

// @ts-expect-error
f2().then(v => v.split(''))
//        ^? (parameter) v: number

Playground

As a result, when constructing the resulting Promise type returned from catch in f1, the result is number | any, which is reduced to any.

Currently, this is a limitation of running outside of --strictNullChecks, but I'm considering whether there are any changes we can make here.
